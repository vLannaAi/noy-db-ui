# Architecture

The noy-db UI family is organised on **two axes** — *what you're looking at* (the components) and *how
it behaves everywhere* (the cross-cutting layers) — and surfaced in **three tiers by how often a
developer touches them**.

```
🛍️  TOP SHELF      components/     ← what you COMPOSE      (open daily)
⚙️   CONFIG         core/           ← what you TUNE         (open once)
🔒  INVARIANT      src/ · internal/ ← what you NEVER open   (the engine)
```

The tiers also map onto the packages: the **INVARIANT** engine (`src/`) is `@noy-db/ui` (this package);
the **TOP SHELF** components, the **CONFIG** `core/` layer, and the `internal/` sub-components live in
each framework **binding** (e.g. `@noy-db/ui-nuxt`).

## 🛍️ Top shelf — the components you compose

The **first level of `components/`** is exactly the 10–15 components you reach for — nothing else.
They map 1:1 to noy-db's domains, so the set is principled, not arbitrary:

| Family | noy-db domain | Components |
|---|---|---|
| **List** | collections | `CollectionList`, `SearchBox`, `SavedSearchMenu`, `RecentSearchMenu`, `NlSearchButton`, `ColumnChooser`, `GroupByControl` |
| **Item** *(soon)* | records | `RecordDetail`, `RecordForm` |
| **Asset** *(soon)* | blobs | `AssetViewer`, `AssetGallery` |
| **Auth** *(soon)* | keyring/sessions | `VaultUnlock`, `KeyringManager` |
| **Owner** *(soon)* | identity & federation | `OwnerCard`, `AccessManager` |

Sub-components a top-shelf component renders for you (header filters, the popover primitive, …) live
in **`internal/`** — explicitly imported, never in your way.

## ⚙️ Config — the layer you tune

`core/` is the **only API you wire up**. Configure it once; every component reads it via a composable,
falling back to safe defaults if you skip it. The cross-cutting layers:

| Layer | Composable | Verdict |
|---|---|---|
| Theme (dark/light) | `useTheme()` | tokens + an attribute |
| i18n (UI strings) | `useNuiI18n()` | inject your `t` |
| AI / LLM | `useLlm()` | inject your client |
| Voice (STT) | `useVoiceInput()` | Web Speech now, pluggable |
| Responsive / touch | `useViewport()` | breakpoint + pointer |
| Zen / focus mode | *(roadmap)* | boolean + CSS |

```ts
// the config layer is realised by each binding — e.g. the Nuxt binding:
import { provideNoydbUi } from '@noy-db/ui-nuxt/core'
provideNoydbUi({ theme: 'system', locale: 'it', llm: myClient /* , voice */ })
```

→ See your binding's **Configuration** docs (e.g. @noy-db/ui-nuxt).

## 🔒 Invariant — the engine you never open

`src/` (the AST search engine, schema adapters, `useCollectionList`, table utils) and `internal/`
(the sub-components) are the **invariant core**. They have no knobs and no domain — you import a couple
of helpers (`schemaFromDescribe`, `useCollectionList`) and otherwise never look inside.

## Why this matters

The layers are **injected, not imported per component**. So adding a *new* top-shelf component
(`Item`, `Asset`, …) inherits theme, i18n, AI, voice and responsive **for free** — you don't re-solve
any of them. That's the whole point: the boring cross-cutting work is done **once**.
